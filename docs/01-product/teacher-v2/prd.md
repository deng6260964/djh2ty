## 1. 文档目标

本版 PRD 用于替代旧版“大而全、按模块并列”的需求组织方式，重新建立更符合老师真实工作节奏的产品方案。

本版重点解决的问题：

- 页面组织散，老师需要频繁跨页面切换
- 课程、反馈、作业、收费没有形成完整闭环
- 首页更像报表页，不像工作推进台
- 收费逻辑不符合老师“提前收费、按课时自动扣减”的实际方式
- 学生端承接弱，信息虽多但不成体系

本版不追求一次性覆盖所有增强功能，而是优先打通老师的高频主链路。

## 2. 产品定位

### 2.1 一句话定位

面向单个老师或小型家教工作室的教学工作台。

### 2.2 产品核心价值

帮助老师每天更顺地完成以下高频任务：

- 看今天要上的课
- 快速排课和调课
- 课后快速补记录
- 统一查看待批改作业
- 基于预收余额自动扣费
- 在余额不足覆盖下一节课前及时提醒收费

### 2.3 非目标

本版暂不追求：

- 多老师协同排班
- 完整 CRM / 销售跟进能力
- 复杂财务对账和发票管理
- 学生端重互动社区化设计
- 考试辅导题库作为主流程能力

## 3. 用户与角色

### 3.1 核心用户

`老师`

特征：

- 单人负责教学、排课、反馈、收费
- 日常事务碎片化，依赖记忆和微信沟通
- 最怕遗漏：课后记录、收费节点、作业跟进

### 3.2 次级用户

`学生 / 家长`

特征：

- 主要是接收信息
- 关注课程、作业、课后反馈、剩余余额
- 当前阶段不承担复杂互动

## 4. 核心问题定义

### 4.1 旧版主要问题

#### 问题一：导航以功能模块为中心，不以工作流为中心

旧版将学生、课程、反馈、进度、收费并列为一级导航，适合罗列能力，不适合推进老师每天的工作。

#### 问题二：课程不是主线

老师真实工作从“这节课”出发，但旧版里排课、课后记录、作业、收费散落在不同页面里，导致高频链路断裂。

#### 问题三：收费模型不匹配

实际场景是“家长预先交钱，系统按课时单价和实际上课时长自动扣减”，旧版更接近传统账单视角，使用不自然。

#### 问题四：首页缺乏推进性

老师登录后最需要知道的是：

- 今天有哪些课
- 哪些课上完还没补记录
- 哪些学生该收费
- 哪些作业待批改

旧版首页偏展示，不够行动导向。

### 4.2 本版重构目标

本版要把产品从“功能集合”重构成“工作流工具”。

## 5. 产品原则

### 5.1 先流程，后功能

所有模块必须嵌入老师真实动作链路，不再设计孤立页面。

### 5.2 首页是工作台，不是报表页

首页第一屏优先推进待办，而不是堆数据卡。

### 5.3 课程是业务主线

排课、上课、课后记录、作业、扣费、收费提醒全部围绕课程发生。

### 5.4 学生是第二入口

学生页承担“看全貌”和长期复盘，不承担最高频操作。

### 5.5 收费是余额管理，不是账单管理

核心对象不是账单，而是学生账户余额、扣费记录和收费提醒。

### 5.6 学生端先做承接

学生/家长端优先承接课程、作业、反馈、余额信息，以轻互动为主。

## 9. 核心业务流程

### 9.1 总流程

老师高频主链路：

`工作台查看今日任务 -> 进入课程 -> 上课 -> 补课后记录 -> 可选布置作业 -> 自动扣减余额 -> 判断是否触发收费提醒`

### 9.2 流程一：排课

目标：

- 快速安排课程
- 避免时间冲突
- 明确课前收费风险

流程：

1. 老师从工作台或课程页点击“快速排课”
2. 选择学生、科目、日期、时间、计划课时
3. 系统实时做时间冲突校验
4. 系统同步检查该学生余额是否足以覆盖下一节课
5. 保存成功后课程进入“待上课”

规则：

- 时间冲突必须在保存前拦截
- 若余额不足，不阻止排课，但需标记“待收费”

### 9.3 流程二：上课后补记录

目标：

- 让课后记录成为课程完成的必要环节

流程：

1. 课程结束后，系统将课程标记为“已上课待记录”
2. 工作台和课程页同步出现“待补记录”
3. 老师进入课程详情，填写最小必要字段
4. 老师确认实际课时
5. 老师可选布置作业
6. 点击“保存并完成课程”

规则：

- 没有完成课后记录，课程不能进入“已完成”
- 老师可暂存记录，后续继续补

### 9.4 流程三：自动扣费

目标：

- 将扣费自然嵌入课程完成动作

公式：

`扣费金额 = 实际课时 × 当前科目课时单价`

流程：

1. 老师完成课程
2. 系统读取科目单价与实际课时
3. 自动生成扣费记录
4. 自动从学生账户余额中扣减
5. 记录扣费后余额

规则：

- 扣费动作必须可追溯
- 单价以课程完成时的有效单价为准
- 若余额不足以覆盖本次扣费，允许形成负余额，但必须高亮风险

### 9.5 流程四：收费提醒

目标：

- 提前一节课提醒老师收费

触发规则：

- 若当前余额不足覆盖“下一节已排课程预计费用”
- 则该学生进入“待收费提醒”

展示位置：

- 工作台待收费模块
- 学生详情账户 Tab
- 课程详情余额区

处理动作：

1. 老师录入收款
2. 系统增加学生账户余额
3. 自动解除风险提示

### 9.6 流程五：学生端承接

目标：

- 将老师端的关键动作自然同步给学生/家长端

同步内容：

- 已排课程
- 已完成课程的课后反馈
- 新布置作业
- 已批改作业结果
- 余额和最近扣费
- 已分享资料

## 12. 关键交互规则

### 12.1 工作台优先级规则

展示优先级：

1. 待补记录
2. 今日课程
3. 待收费提醒
4. 待批改作业

说明：

“课已上但未闭环”优先级最高。

### 12.2 风险标签规则

系统自动识别以下轻量风险：

- 余额不足覆盖下一节课
- 连续两次未交作业
- 最近两周无课
- 存在待补记录

### 12.3 空状态规则

- 工作台无课时突出“快速排课”
- 作业为空时突出“从课程中布置作业”
- 资料为空时突出“上传并分享第一份资料”

### 12.4 异常状态规则

- 请求失败时显示清晰错误态和重试入口
- 课程完成前若扣费失败，需提示并保留待处理状态
- 任何涉及余额变动的操作都必须有结果回执

## 14. 非功能需求

### 14.1 性能

- 工作台首屏加载目标 <= 2 秒
- 课程详情打开 <= 1 秒
- 完成课程动作反馈 <= 2 秒

### 14.2 易用性

- 高频动作不超过 3 次点击可达
- 课程完成闭环尽量在单页内完成

### 14.3 可追溯性

- 所有余额变化必须保留记录
- 所有课程状态变化必须可追踪

## 15. MVP 验收标准

### 15.1 老师端

- 老师可从工作台看到今日课程、待补记录、待收费提醒、待批改作业
- 老师可从课程详情完成课后记录并自动扣费
- 系统可基于余额判断“下一节课是否需要收费”
- 老师可在学生账户中录入收款并解除收费风险

### 15.2 学生端

- 学生/家长可查看课程、作业、反馈、资料、余额
- 学生端内容与老师端动作同步一致
