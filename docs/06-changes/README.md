# 变更记录目录

> 状态：当前
> 范围：全项目
> 更新：2026-04-27

本目录用于记录已经发生的产品、设计、架构、接口、数据结构和重要实现变更。它回答的是：“这次到底改了什么、为什么改、影响哪里、如何验证、后续要注意什么。”

## 与其他目录的区别

| 目录 | 记录对象 | 主要用途 |
| --- | --- | --- |
| `docs/04-implementation/` | 计划、任务拆解、测试用例、验证记录 | 推进实现和交接工作 |
| `docs/05-ai-worklogs/` | AI 会话中提炼出的过程结论 | 保留 vibecoding 上下文 |
| `docs/06-changes/` | 已经完成或已经确认的变更 | 追溯变更、辅助多人并行协作 |

## 什么时候需要新增变更记录

出现以下任一情况时，应新增一条 change 记录：

- 产品行为发生变化
- 页面结构、交互路径或信息架构发生变化
- API 入参、返回值、状态码或权限规则发生变化
- 数据模型、余额规则、扣费规则等业务规则发生变化
- 跨端协作点发生变化，例如后端接口影响管理端和学生端
- 代码重构改变了模块边界、公共工具或约定
- 修复了高风险 bug，需要后续研发知道原因和影响范围

小的样式修正、文案微调、无行为变化的内部整理，一般不需要单独记录。

## 命名规则

按年月分组，文件名使用日期加主题：

```text
docs/06-changes/YYYY-MM/YYYY-MM-DD-topic.md
```

示例：

```text
docs/06-changes/2026-04/2026-04-27-doc-structure-cleanup.md
docs/06-changes/2026-04/2026-04-27-teacher-v2-billing-rule.md
```

## 写入规则

1. 一条记录只描述一个清晰变更。
2. 记录已经确认或已经完成的事实，不记录未经确认的设想。
3. 必须写清楚影响范围和验证方式。
4. 如果变更更新了当前事实源，应在记录中列出对应文档。
5. 如果变更来自某个 workstream 或 AI 工作日志，应互相引用。
